### Cross-Plan Concern: Stale Hit Rate on Theme Selection (checker W2)

After a round completes and the user navigates back to theme selection via "Sair" or "Voltar", the hit rate labels on the theme selection screen still show pre-round values. This is because `ThemeSelectionController` calls `view.updateHitRate()` only once during `initialize()` at startup, but StatsService data has been updated by the round.

**Recommended fix (implements checker W2 fix hint):**
1. Add `public void refreshHitRates()` to `ThemeSelectionController` that re-reads hit rates from `StatsService.getHitRate(themeName)` for each theme and calls `view.updateHitRate()` — same logic as the startup loop in `initialize()`.
2. In `StudyRoundController.handleExit()` and `showRoundComplete()`, call the refresh method before switching back to "themeSelection" scene. Since StudyRoundController doesn't hold a reference to ThemeSelectionController, this requires either: (a) passing a `Runnable` callback from ThemeSelectionController, or (b) using the ScreenController to look up a registered refresh handler.
3. **Recommended approach (lightest coupling):** ThemeSelectionController registers a `consumer` in ScreenController or sets a static/instance callback. Or simpler: the Relatórios button handler (Plan 04) can call both `reportsController.refresh()` AND re-create the hit rate labels by re-invoking the hit rate loop on ThemeSelectionController.

**Implementation owner:** Plan 04 (Theme + Main Wiring) + Plan 02 (Round Integration).
